  <behavior>
    - Reconnect: POST with name="Alice", no cookie, Alice is in participants hash → returns { participantId: "alice-uuid", name: "Alice" }, re-issues cookie with same options as original join (httpOnly, secure, sameSite: strict, maxAge: 86400, path: "/")
    - New joiner duplicate name: POST with name="Alice", no cookie, Alice is in participants hash, but name did NOT match (name-match path not applicable) → actually the name-match path IS the only path for "name already in room" — any no-cookie request with a taken name is treated as reconnect.     - New participant: POST with name="Bob", no cookie, Bob is NOT in participants hash → creates new UUID, calls joinRoom, sets cookie, returns 200. Existing behavior unchanged.
    - Double Redis call avoidance: `getParticipants` called ONCE (before branching) when no cookie is present, result reused in both name-match check and new-participant path.
    - Reconnect does NOT call joinRoom() — existing record is reused as-is, no data mutation.
    - Reconnect does NOT trigger Pusher broadcast — participant is already in the list.
  </behavior>
  <action>
    Rewrite the section of `app/api/rooms/[roomId]/join/route.ts` after the existing cookie check (Path A) and before the new-participant path. The new structure:

    ```
    [Path A] existing cookie → verify in Redis → return existing participantId (UNCHANGED)
    [Path B] no cookie → call getParticipants ONCE → find name match
      [B1] match found → re-issue cookie with match.participantId, return { participantId: match.participantId, name: match.name }, NO joinRoom call, NO Pusher trigger
      [B2] no match → new participant path (existing: crypto.randomUUID, joinRoom, set cookie, Pusher trigger, return)
    ```

    Cookie re-issuance options for B1 must be identical to the original cookie set in B2:
    `{ httpOnly: true, secure: process.env.NODE_ENV === 'production', sameSite: 'strict', maxAge: 24 * 60 * 60, path: '/' }`

    Per RESEARCH.md: name comparison uses `===` (both sides trimmed by joinSchema at input time, and stored names were trimmed at join time). No case folding.

    Do NOT call joinRoom() in Path B1. Do NOT trigger pusherServer.trigger() in Path B1. These are explicit anti-patterns from RESEARCH.md.

    IMPORTANT: Note from RESEARCH.md — the design intent is that a no-cookie request with a name already in the room IS the reconnect path. There is no separate "new person trying to steal name" rejection because: if the name is in the room, they ARE treated as the reconnecting person. This is the accepted tradeoff (teams are trusted). Do not add an extra 409 branch — the CONTEXT.md decision says uniqueness is enforced at join time, and Path B IS that enforcement (returning existing UUID, not creating new).
